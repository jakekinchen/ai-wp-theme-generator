# Architecture Decisions

## AI Generates ThemePlan, Not Raw Files

The model returns constrained design data. The app compiles every WordPress file deterministically. This keeps file paths, block names, PHP headers, and validation under application control.

## Deterministic Compiler Owns Block Markup

WordPress block markup is generated by local renderers for known recipes. The compiler is easier to test than model-authored markup and prevents invented blocks from entering the zip.

## Small Core Block Allowlist

The MVP supports common core blocks only: template parts, patterns, layout blocks, headings, paragraphs, buttons, query loops, post fields, site title, navigation, and page list. Custom HTML and Classic/freeform blocks are rejected.

## `theme.json` Handles Styling

Design tokens are rendered into `theme.json` version 3. Custom CSS is limited to small structural enhancements.

## No Remote Assets

Generated output does not load remote fonts, scripts, styles, images, or iframes. This keeps the first MVP portable and safer to inspect.

## Structural Preview

The browser preview renders from the same `ThemePlan`, but it does not pretend to be WordPress. A live WordPress preview would require a disposable WordPress runtime and activation pipeline.

## Optional `wp-env` Smoke Test

Docker-backed WordPress activation is useful reviewer confidence but should not block normal tests. The optional hook exists for future expansion.

## Alternatives Rejected

- Letting the AI generate block HTML directly: too fragile and hard to secure.
- Letting the AI generate a zip: hides file-level validation.
- Remote font loading: adds network dependency and review risk.
- Live WordPress preview for MVP: high complexity compared with deterministic compile and validation.

## Security Considerations

Inputs are normalized before AI, generated plans are schema-validated, compiled files are checked for unsafe paths and blocks, PHP is limited to pattern headers, and packaging refuses invalid output.
